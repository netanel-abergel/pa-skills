// what is an AI PA?
An AI Personal Assistant is not a chatbot.
A chatbot waits for you to ask something. An AI PA works for you. It manages your calendar, monitors your systems, coordinates with other agents, sends your morning briefing, and handles tasks end-to-end — without needing hand-holding.

Heleni is a production AI PA built on OpenClaw, an open-source agent runtime. She runs 24/7, connected to WhatsApp, Google Calendar, and monday.com. She schedules meetings by talking to other PAs, detects billing failures before they cascade, and delivers daily briefings at 7:30 AM — all autonomously.
// why these skills exist
Every skill here was born from a real problem.
This isn't a theoretical skill library. Every SKILL.md in this repo was created because Heleni needed it in production. The billing-monitor was built after 5+ PAs went down from undetected API billing errors in a single day. The self-learning skill exists because Heleni kept repeating the same mistakes across sessions — corrections weren't persisting. The memory-tiering system was designed when daily memory files grew past 200 lines and started wasting the context window.

These skills represent months of iteration with a real AI PA serving a real person — scheduling meetings, managing a network of 15+ peer PAs, handling WhatsApp groups, and running autonomous cron jobs. The patterns are hard-won.
// how heleni thinks
Execution model, not conversation model.
Heleni's behavior is defined in a SOUL.md file — a behavioral contract that shapes how she operates. The core principles:

Execute, then report. Never narrate what you're about to do. Never share intermediate steps. Never list pros and cons when one clear answer exists. Do the thing, then say what happened.

Silence is correct behavior. In group chats, Heleni only speaks when directly addressed or when she has clear responsibility. No reactions, no echoing, no noise. When there's nothing valuable to add, the right move is silence.

Proactive over reactive. Before ending any task, Heleni asks herself: "Is there anything the owner would want to know that I haven't surfaced?" She monitors for calendar conflicts, unanswered messages, billing issues, and system health — and alerts without being asked.

Permission boundaries. Reading data, organizing files, updating memory — free to do. Sending messages to people, making purchases, deleting external data — always ask first. The line is: internal and reversible = execute; external or irreversible = confirm.

Key Learnings from Production
// these are real corrections and patterns discovered while running Heleni in production — each one caused a bug, outage, or user frustration before it became a rule
🎯
Execution > Conversation
Execute first, report results only. Never share reasoning, intermediate steps, or "let me check" narration. Lead with one recommendation instead of listing options.
Why: The owner explicitly corrected this pattern. Showing search process and multiple options added friction without value. Results matter, process doesn't.
🔒
Credential Exposure Protocol
If credentials appear in chat, warn immediately on receipt — before any operations. Then proceed with task, and remind to revoke after completion.
Why: GitHub tokens were shared in WhatsApp. The PA used them first and warned after — too late. The security warning must come before any action.
📡
Explicit Delivery Targets
Cron jobs without explicit delivery targets fall back to last active session — can leak messages to wrong chats. Always pin the target explicitly.
Why: A daily AI digest intended for a work group was delivered to the owner's private DM because the cron job used session context instead of a pinned target.
🧠
Memory Discipline
Daily memory files should stay under 50 lines. Extract only durable facts: contacts, decisions, persistent rules, unresolved blockers. Not event logs.
Why: A daily notes file grew to 200+ lines with transient events, eating context window for no value. Most content was stale within hours.
🔍
Runtime Identity Check
Model identity must be checked via live session status, never assumed from context. Models can change mid-session without notice.
Why: Model switched from GPT to Claude mid-session. The PA kept reporting the old model because it relied on compacted context instead of checking runtime state.
💳
Proactive Billing Alerts
Billing errors should trigger automatic fallback + notification, not reactive handling. Install billing-monitor with heartbeat integration for periodic health checks.
Why: 5+ PAs went down simultaneously from billing errors. Each required manual detection and intervention. Now automated with the billing-monitor skill.
📋
Config Deduplication
SOUL.md, AGENTS.md, and MEMORY.md had 60% content overlap causing contradictions. Keep one source of truth per concern — facts vs behavior vs identity.
Why: Conflicting instructions across three config files caused inconsistent PA behavior. Merged into clear separation: SOUL = behavior, MEMORY = facts, IDENTITY = who.
📱
Connected != Working
"Connected and listening" with Messages=0 means the ingest layer is broken, not the messaging platform. Escalate as platform issue, not user config.
Why: Multiple PAs showed WhatsApp as connected but never responded. The problem wasn't WhatsApp — it was the binding between the channel and the agent runtime.
